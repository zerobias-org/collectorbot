# Collector Bot Package Structure Rules

This document defines the rules and conventions for collector bot packages in the **zerobias-org/collectorbot** repository. Claude Code MUST follow these rules when developing, modifying, or verifying collector bot code.

## ⚠️ ZeroBias-Org Repository - Key Differences

**This is the community zerobias-org repository, NOT the proprietary auditlogic repository.**

**Critical Scope Differences:**

| Package Type | Scope in THIS Repo | Notes |
|-------------|-------------------|-------|
| Collectors | `@zerobias-org/collectorbot-*` | Different from auditlogic |
| Modules | `@zerobias-org/module-*` | Prefer zerobias-org, fallback to auditlogic |
| Products | `@zerobias-org/product-*` | ALWAYS zerobias-org, auditlogic deprecated |
| Schemas | `@auditlogic/schema-*` | Same as auditlogic (centralized) |
| Infrastructure | `@auditmation/*` | Same as auditlogic |

**Package Name Derivation:**
- Directory: `package/<vendor>/<suite?>/<product>/`
- Package: `@zerobias-org/collectorbot-<vendor>-<suite?>-<product>`
- Example: `package/avigilon/alta/access/` → `@zerobias-org/collectorbot-avigilon-alta-access`

**Product Discovery:**
```bash
npm view @zerobias-org/product-bundle --json | jq '.dependencies'
```

## Package Structure Overview

Collector bot packages follow a hierarchical naming convention:

```
package/<vendor>/<suite-if-exists>/<product>/
```

**Rules:**
- `<vendor>` - Always required (e.g., `amazon`, `microsoft`, `google`, `adp`)
- `<suite>` - Optional, only when the vendor has product suites (e.g., `aws` for Amazon, `365` for Microsoft)
- `<product>` - The specific product being collected from (e.g., `iam`, `ec2`, `entra`)

**Examples:**
- `package/amazon/aws/iam/` - AWS IAM collector (vendor=amazon, suite=aws, product=iam)
- `package/amazon/aws/ec2/` - AWS EC2 collector (vendor=amazon, suite=aws, product=ec2)
- `package/microsoft/entra/` - Microsoft Entra collector (vendor=microsoft, no suite, product=entra)
- `package/adp/workforcenowng/` - ADP Workforce Now NG collector (vendor=adp, no suite, product=workforcenowng)
- `package/google/workspace/sheets/` - Google Workspace Sheets (vendor=google, suite=workspace, product=sheets)

**IMPORTANT:** The legacy `collector/` directory at the repository root is **deprecated**. All new collectors MUST be created in the `package/` directory structure.

### Complete File Structure

```
package/<vendor>/<suite-if-exists>/<product>/
├── .eslintrc                    # ESLint configuration
├── .gitignore                   # Git ignore rules
├── .mocharc.json               # Mocha test configuration
├── .npmrc -> ../../../../.npmrc # NPM config (symlink to root)
├── .nvmrc -> ../../../../.nvmrc # Node version (symlink to root)
├── CHANGELOG.md                # Auto-generated by Lerna (DO NOT EDIT)
├── collector.yml               # Collector metadata - schema classes
├── hub.yml                     # Hub module references
├── package.json                # NPM package configuration
├── npm-shrinkwrap.json         # Dependency lock file
├── parameters.yml              # Collector input parameters (OpenAPI)
├── tsconfig.json               # TypeScript configuration
├── README.md                   # (optional) Package documentation
├── USERGUIDE.md               # (optional) User guide
├── service-map.yml            # (optional) Service type mappings
├── product-map.yml            # (optional) Product mappings
│
├── generated/                  # Auto-generated code (DO NOT EDIT)
│   ├── BaseClient.ts          # Base client class with injected modules
│   ├── index.ts               # Re-exports all generated modules
│   ├── inversify.config.ts    # Dependency injection container setup
│   ├── types.ts               # Symbol types for DI
│   └── model/
│       ├── index.ts           # Model exports

│
├── src/                        # Source code (EDITABLE)

│   ├── Util.ts / utils.ts     # (optional) Utility functions

│   └── Ensurers.ts            # (optional) Data validation/transformation
│
└── test/
    └── integration/
        └── Collector*IT.ts    # Integration tests
```

## Package Naming Conventions

### NPM Package Scopes

**IMPORTANT:** This is the **zerobias-org/collectorbot** repository. Package scopes differ from the proprietary auditlogic repository.

Different package types use different NPM scopes:

| Package Type | Scope (Priority) | Fallback Scope | Example |
|-------------|-----------------|----------------|---------|
| **Collector bots** (what we develop) | `@zerobias-org/collectorbot-*` | - | `@zerobias-org/collectorbot-avigilon-alta-access` |
| **Hub modules** | `@zerobias-org/module-*` | `@auditlogic/module-*` | `@zerobias-org/module-avigilon-alta-access` |
| **Product packages** | `@zerobias-org/product-*` | - | `@zerobias-org/product-avigilon-alta-access` |
| **Schema packages** | `@auditlogic/schema-*` | - | `@auditlogic/schema-avigilon-alta-access` |
| **Schema TypeScript types** | `@auditlogic/schema-*-ts` | - | `@auditlogic/schema-avigilon-alta-access-ts` |
| **Hub infrastructure** | `@auditmation/hub-*` | - | `@auditmation/hub-client` |
| **Utility packages** | `@auditmation/util-*` | - | `@auditmation/util-collector-utils` |
| **Type packages** | `@auditmation/types-*` | - | `@auditmation/types-core-js` |

**Dependency Priority Rules:**
1. **Modules:** Prefer `@zerobias-org/module-*`, fallback to `@auditlogic/module-*` if not available
2. **Products:** Always use `@zerobias-org/product-*` (auditlogic products are deprecated for this repo)
3. **Schemas:** Always use `@auditlogic/schema-*` (no alternative, centralized schema management)
4. **Infrastructure:** Always use `@auditmation/*` (hub, utils, types - no alternatives)

### Product Packages (Transitive Dependencies)

**Product packages** provide product metadata and UUIDs:
- **Naming:** `@zerobias-org/product-<vendor>-<suite?>-<product>`
- **Examples:**
  - `@zerobias-org/product-avigilon-alta-access` (vendor=avigilon, suite=alta, product=access)
  - `@zerobias-org/product-github-github` (vendor=github, no suite, product=github)
  - `@zerobias-org/product-amazon-aws-iam` (vendor=amazon, suite=aws, product=iam)
- **Note:** Suite is included with hyphen when present!
- **NOT added to dependencies** - module already depends on it
- **Used for:** Finding product UUID for `hub.yml`

**Discovering Available Products:**

To find all available product packages in the ZeroBias ecosystem:

```bash
npm view @zerobias-org/product-bundle --json | jq '.dependencies'
```

This will list all products following the pattern `@zerobias-org/product-<vendor>-<suite?>-<product>`.

Match the pattern to your collector's directory structure:
- Directory: `package/avigilon/alta/access/` → Product: `@zerobias-org/product-avigilon-alta-access`
- Directory: `package/github/github/` → Product: `@zerobias-org/product-github-github`

### Schema Packages: Base vs TypeScript (-ts)

**CRITICAL:** Understand the difference between base schema packages and their TypeScript counterparts:

| Package | Purpose | Usage |
|---------|---------|-------|
| `@auditlogic/schema-<name>` | Base schema definition | **Must be in dependencies** - required for runtime system installation even if not imported directly in code |
| `@auditlogic/schema-<name>-ts` | Generated TypeScript interfaces | **Import in code** - use for type safety when creating batch items and working with schema data types |

**Example in package.json:**
```json
{
  "dependencies": {
    "@auditlogic/schema-amazon-aws-iam": "^1.16.2",      // Base schema (runtime dependency)
    "@auditlogic/schema-amazon-aws-iam-ts": "^1.16.2"   // TypeScript types (for imports)
  }
}
```

**Example in Mappers.ts:**
```typescript
// Import TypeScript types from -ts package for type safety
import { AwsIamUser, AwsIamRole } from '@auditlogic/schema-amazon-aws-iam-ts';

export function toAwsUser(...): AwsIamUser {
  return {
    id: ...,
    name: ...,
  };
}
```

## Root Configuration Files

### `.mocharc.json`

**Purpose:** Mocha test runner configuration

**Rules:**
- File extension must be `ts`
- Requires `ts-node/register` for TypeScript support
- Timeout set to 6000000ms (100 minutes) for long-running integration tests

**Template:**
```json
{
  "extension": ["ts"],
  "require": "ts-node/register",
  "timeout": 6000000
}
```

**Validation:**
- [ ] Must be valid JSON
- [ ] Extension must include "ts"
- [ ] Timeout must be exactly 6000000 (100 minutes standard)

---

### `tsconfig.json`

**Purpose:** TypeScript compiler configuration

**Rules:**
- Target ES5 for compatibility
- Module system: CommonJS
- Output directory: `dist/`
- Includes: `src/**/*`, `test/**/*`, `generated/**/*`
- Excludes: `dist`, `node_modules`
- Decorators must be enabled (for inversify DI)

**Template:**
```json
{
  "compilerOptions": {
    "module": "commonjs",
    "noImplicitAny": false,
    "suppressImplicitAnyIndexErrors": true,
    "target": "ES5",
    "strict": true,
    "useUnknownInCatchVariables": false,
    "moduleResolution": "node",
    "removeComments": true,
    "sourceMap": true,
    "noLib": false,
    "allowJs": true,
    "declaration": true,
    "lib": ["es6", "es5"],
    "outDir": "dist",
    "typeRoots": ["node_modules/@types"],
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true
  },
  "include": [
    "src/**/*",
    "test/**/*",
    "generated/**/*"
  ],
  "exclude": [
    "dist",
    "node_modules"
  ]
}
```

**Validation:**
- [ ] `experimentalDecorators` must be `true`
- [ ] `emitDecoratorMetadata` must be `true`
- [ ] `outDir` must be `dist`
- [ ] Must include `generated/**/*` in includes

### `.gitignore`

**Purpose:** Git ignore patterns

**Template:**
```
node_modules/
dist/
generated/
```

**Rules:**
- `generated/` must be ignored (auto-generated by `npm run generate`)
- `dist/` must be ignored (build output)
- `node_modules/` must be ignored (npm packages)
- **NEVER include** `.npmrc` or `.nvmrc` (they are symlinks)

---

### `.npmrc` and `.nvmrc`

**Purpose:** NPM and Node.js version configuration

**Rules:**
- **MUST be symlinks** to root configuration - NEVER copy these files
- Ensures consistent npm/node settings across all packages
- Symlink depth depends on package location in the hierarchy

**Creating Symlinks:**

For a package at `package/<vendor>/<suite>/<product>/` (4 levels deep):
```bash
cd package/<vendor>/<suite>/<product>/
ln -s ../../../../.npmrc .npmrc
ln -s ../../../../.nvmrc .nvmrc
```

For a package at `package/<vendor>/<product>/` (3 levels deep):
```bash
cd package/<vendor>/<product>/
ln -s ../../../.npmrc .npmrc
ln -s ../../../.nvmrc .nvmrc
```

**Verification:**
```bash
ls -la .npmrc .nvmrc
# Should show -> pointing to root files, not regular files
```

## Metadata Files

### `collector.yml`

**Purpose:** Defines which schema classes this collector populates

**Structure:**
```yaml
classes:
    # Reference to schema package and its classes
    - "@auditlogic/schema-<vendor>-<product>":
          - ClassName1
          - ClassName2
    - "@auditlogic/schema-<vendor>-<product>-<subproduct>":
          - ClassName3
```

**Rules:**
- `classes` is the root key
- Each entry is a schema package name (NPM scope: `@auditlogic/schema-*`)
- Under each package, list all schema classes this collector creates/updates
- Schema package names must match dependencies in `package.json`
- Comments can be added with `#`

**Example (AWS IAM):**
```yaml
classes:
    - "@auditlogic/schema-amazon-aws":
          - AwsPolicyStatement
    - "@auditlogic/schema-amazon-aws-iam":
          - AwsIam
          - AwsIamCustomerPolicy
          - AwsIamGroup
          - AwsIamIdentityProvider
          - AwsIamInlinePolicy
          - AwsIamRole
          - AwsIamUser
```

**Example (Empty/Generic):**
```yaml
classes:
    # The classes should be defined in the input parameters
```

**Validation:**
- [ ] Must have `classes` root key
- [ ] All referenced schema packages must exist in `package.json` dependencies
- [ ] Class names should match actual schema class exports

---

### `hub.yml`

**Purpose:** Defines Hub modules and products this collector uses

**Structure:**
```yaml
modules:
  '@auditlogic/module-<vendor>-<product>':
    id: <uuid>                    # Module UUID
    name: module-<vendor>-<product>
    module: <ModuleName>          # Module class name
    version: <semver>             # Module version
    operations:                   # Operations used by this collector
      operationName: <operation-uuid>
    permissions:                  # Required permissions
      - permission:action

products:
  <vendor>.<product>.<subproduct>:
    id: <uuid>                    # Product UUID
    name: <Human Readable Name>
```

**Rules:**
- `modules` section lists all Hub modules the collector connects to
- `products` section identifies the product being collected from
- Module IDs are UUIDs (get from module package or platform)
- **Operations are OPTIONAL** - can be omitted entirely
- Permissions list required access rights (helpful for documentation)

**Example (AWS IAM):**
```yaml
modules:
  '@auditlogic/module-amazon-aws-iam':
    id: d68b3d81-3a47-417c-8485-397dc602380b
    name: module-amazon-aws-iam
    module: Iam
    version: 7.0.4
    operations:
      listPolicies: 1c4e0368-5c94-50da-a47f-b425b4ef8c56
      getPolicyVersion: 6eb4d40e-6709-5483-8ffb-06f483af59e0
      listRoles: 8780c795-d255-57f0-b186-89e3c2dd0f09
    permissions:
      - iam:ListPolicies
      - iam:GetPolicyVersion
      - iam:ListRoles

products:
  amazon.aws.iam:
    id: 98926f9a-d2a6-5316-aa4d-44169af7123c
    name: AWS Identity and Access Management
```

**Example (Empty/Generic):**
```yaml
modules:  {}
products: {}
```

**Validation:**
- [ ] Must have both `modules` and `products` keys (can be empty `{}`)
- [ ] Module package names must match dependencies in `package.json`
- [ ] UUIDs must be valid format

### `parameters.yml`

**Purpose:** OpenAPI 3.0 schema defining collector input parameters

**Structure:**
```yaml
openapi: 3.0.3
info:
  description: Stub to use for codegen of parameters for a hub client
  version: ''
  title: '@auditmation/hub-client-codegen'
paths:
  /foo:
    get:
      tags:
        - foo
      operationId: getFoo
      responses:
        '204':
          description: Foo
components:
  schemas:
    Parameters:
      type: object
      # Optional: required fields
      required:
        - fieldName
      properties:
        fieldName:
          type: string
          description: Field description
```

**Rules:**
- Must be valid OpenAPI 3.0.3 specification
- `components.schemas.Parameters` defines the input parameters object
- The `paths` section is a stub for codegen compatibility
- Parameters can have:
  - `type`: string, number, boolean, array, object
  - `description`: Human-readable description
  - `required`: List of required field names
  - `format`: e.g., `uuid` for UUID fields
  - `default`: Default value
  - `example`: Example value
  - `items`: For arrays, defines item schema
  - `$ref`: Reference to other schemas

**Example (Simple - no parameters):**
```yaml
openapi: 3.0.3
info:
  description: Stub to use for codegen of parameters for a hub client
  version: ""
  title: '@auditmation/hub-client-codegen'
paths:
  /foo:
    get:
      tags:
        - foo
      operationId: getFoo
      responses:
        '204':
          description: Foo
components:
  schemas:
    Parameters:
      type: object
```

**Example (With parameters):**
```yaml
openapi: 3.0.3
info:
  description: Stub to use for codegen of parameters for a hub client
  version: ''
  title: '@auditmation/hub-client-codegen'
paths:
  /foo:
    get:
      tags:
        - foo
      operationId: getFoo
      responses:
        '204':
          description: Foo
components:
  schemas:
    Parameters:
      type: object
      required:
        - targetId
      properties:
        targetId:
          description: ID of the target to execute against
          type: string
          format: uuid
        skipServiceTypes:
          type: array
          description: List of service types to skip
          items:
            type: string
          example: ["s3", "ce"]
```

**Validation:**
- [ ] Must be valid OpenAPI 3.0.3
- [ ] Must have `components.schemas.Parameters`
- [ ] `Parameters` must be type `object`

## Package Configuration

### `package.json`

**Purpose:** NPM package configuration

**Naming Convention:**
```
@zerobias-org/collectorbot-<vendor>-<suite?>-<product>
```

**How to Determine Package Name from Directory:**

The package name is derived directly from the directory structure:
- Directory: `package/<vendor>/<suite?>/<product>/`
- Package: `@zerobias-org/collectorbot-<vendor>-<suite?>-<product>`

**Examples:**
- `package/avigilon/alta/access/` → `@zerobias-org/collectorbot-avigilon-alta-access`
- `package/github/github/` → `@zerobias-org/collectorbot-github-github`
- `package/auditmation/generic/tls/` → `@zerobias-org/collectorbot-auditmation-generic-tls`

**Note:** When suite is present, it's included with hyphen: `<vendor>-<suite>-<product>`

**Required Fields:**

| Field | Description | Example |
|-------|-------------|---------|
| `name` | Package name with scope | `@zerobias-org/collectorbot-avigilon-alta-access` |
| `version` | Semantic version | `1.1.2` |
| `description` | Short description | `Avigilon Alta Access Collectorbot` |
| `license` | License type | `UNLICENSED` |
| `author` | Author email | `ctamas@zerobias.com` |
| `main` | Entry point | `dist/src/index.js` |
| `directories.src` | Source directory | `src` |
| `files` | Published files | `["dist", "pnpm-lock.yaml", "*.yml", "*.md"]` |

**Required Dependencies Categories:**

1. **Module Dependencies** (`@zerobias-org/module-*` or `@auditlogic/module-*`):
   - Hub modules this collector uses
   - Must match entries in `hub.yml`
   - **Prefer** `@zerobias-org/module-*`, fallback to `@auditlogic/module-*`
   - Use `^` version range

2. **Product Dependencies** (`@zerobias-org/product-*`):
   - Product packages for product metadata
   - Must match the product for this collector
   - Always use `@zerobias-org/product-*` scope
   - Use `latest` version

3. **Schema Dependencies** (`@auditlogic/schema-*`):
   - Schema packages for data types
   - Both base schema and TypeScript types (`*-ts`)
   - Must match entries in `collector.yml`
   - Always use `@auditlogic/schema-*` (centralized schema management)
   - Use `^` version range or `latest`

4. **Core Dependencies:**
   - `@auditmation/hub-client` - Hub client base
   - `@auditmation/hub-core` - Hub core types
   - `@auditmation/util-collector-utils` - Collector utilities
   - `@auditmation/module-auditmation-auditmation-hub` - Hub module (always included)
   - `@auditmation/module-auditmation-auditmation-platform` - Platform module (always included)
   - `@auditmation/types-core-js` - Core type definitions
   - `inversify` - Dependency injection
   - `reflect-metadata` - Decorator metadata

5. **What NOT to Include:**
   - **Vendor packages** - Not used in zerobias-org collectors

## Error Handling

### Error Types and When to Use

| Situation | Action | Effect |
|-----------|--------|--------|
| Connection failure, auth error, module unavailable | `throw new Error(...)` | Fails entire collector run |
| API operation fails for all items | `throw new Error(...)` | Fails entire collector run |
| **Pipeline-level problem** (access issue, operation failure) | `throw new Error(...)` | Fails entire collector run |
| **Batch item problem** (mapping issue, missing property, broken link) | `await batch.error(message, data)` | Logs error, continues collection, batch marked red |
| **Data quality issue** (missing optional field, unexpected value) | `await batch.warn(message, data)` | Logs warning, continues collection |

**General Rule:**
- **Error**: Problem with the data/pipeline that affects collection quality
- **Warn**: Data quality issue that doesn't prevent collection

### Error Handling Pattern

```typescript
async run(): Promise<void> {
  // Initialization errors should throw
  try {
    await this.init();
  } catch (error) {
    throw new Error(`Failed to initialize: ${error.message}`);
  }

  // Collection continues on item-level errors
  const batch = await this.initBatchForClass(MyClass);

  await items.forEach(async (item) => {
    try {
      if (!isValid(item)) {
        await batch.warn('Item missing optional data', item);
        return;
      }
      const mapped = toSchemaClass(item);
      await batch.add(mapped, item);
    } catch (err) {
      await batch.error(`Failed to process item: ${err.message}`, item);
    }
  });

  await batch.end();
}
```

## Preview Mode

### Handling Preview Mode

Preview mode limits collection for testing/debugging. **All collectors MUST support preview mode.**

```typescript
@injectable()
export class CollectorImpl extends BaseClient {
  // Calculate preview count: use context value or default to 10
  private previewCount?: number = this.context.previewMode
    ? (this.context.previewCount || 10)
    : undefined;
```

### Preview Count Rules

| Rule | Description |
|------|-------------|
| **Per-batch limit** | Limit each batch to `previewCount` items, not total |
| **Per-source limit** | If collecting from multiple sources (e.g., 2 orgs), limit per source |
| **Use in forEach** | Pass `previewCount` to `PagedResults.forEach()` as the limit parameter |

**Example - Multiple Sources:**
```typescript
// Collecting users from 2 organizations
// With previewCount=10: collect 10 users from EACH org, not 10 total

for (const org of organizations) {
  const usersBatch = await this.initBatchForClass(User);
  const users = await this.module.getUsers(org.id);

  await users.forEach(async (user) => {
    await usersBatch.add(toUser(user), user);
  }, undefined, this.previewCount);  // Limit per org

  await usersBatch.end();
}
```

## Batch Processing and GroupId

### Understanding Batches

Collectors write data to AuditgraphDB using the `Batch<T>` utility from `@auditmation/util-collector-utils`. Understanding batch behavior is **CRITICAL** for correct data collection.

### The GroupId Concept

**CRITICAL RULE:** The `groupId` determines how batches interact with existing data.

```typescript
const batch = new Batch<SchemaClass>(
  className,        // Schema class name
  this.platform,    // Platform connector
  this.logger,      // Logger
  jobId,            // Job ID
  tags,             // Optional tags
  groupId           // GROUP ID - CRITICAL
);
```

### GroupId Behavior

When two batches have the **same groupId** for the **same class**:
- If Batch A has 30 items and Batch B has 0 items
- Batch B will **DELETE all 30 items** from Batch A

This is by design to handle object removal from source systems (e.g., deleted users, removed access).

### GroupId Rules

| Scenario | GroupId Strategy | Example |
|----------|-----------------|---------|
| Single data source, global scope | Use account/tenant ID | `${accountId}-global` |
| Single data source, regional scope | Use account + region | `${accountId}-${region}` |
| Multiple independent data sources | Use unique identifier per source | `${accountId}-users-ad`, `${accountId}-users-ldap` |
| Different lists of same class | **MUST use different groupIds** | See example below |

### Example: Correct GroupId Usage

```typescript
// CORRECT: Two different user sources use different groupIds
// AD users and LDAP users are independent lists
const adUsersBatch = new Batch<User>('User', ..., `${accountId}-users-ad`);
const ldapUsersBatch = new Batch<User>('User', ..., `${accountId}-users-ldap`);

// CORRECT: Same source, different regions
const usEastBatch = new Batch<Instance>('Instance', ..., `${accountId}-us-east-1`);
const euWestBatch = new Batch<Instance>('Instance', ..., `${accountId}-eu-west-1`);
```

### Example: INCORRECT GroupId Usage (Data Loss!)

```typescript
// WRONG: Same groupId for different data sources
// The second batch will DELETE items from the first!
const adUsersBatch = new Batch<User>('User', ..., `${accountId}-users`);
const ldapUsersBatch = new Batch<User>('User', ..., `${accountId}-users`);
// ⚠️ LDAP batch will remove all AD users!
```

### Common GroupId Patterns

**AWS Collectors:**
```typescript
// Global resources (IAM, S3, etc.)
groupId = `${accountId}-global-${serviceName}`;

// Regional resources (EC2, RDS, etc.)
groupId = `${accountId}-${region}`;
```

**Multi-tenant Collectors:**
```typescript
// Per-tenant grouping
groupId = `${tenantId}-${resourceType}`;
```

**Best Practice:**
- Always think: "What defines a complete, independent set of this data?"
- When in doubt, use more specific groupIds
- Document your groupId strategy in the collector
